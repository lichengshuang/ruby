Ruby异常处理(Exception)

Ruby提供了一个很好的机制来处理异常。我们附上的代码可以在开始/结束块引发的异常，并使用救援条款告诉Ruby我们要处理的异常类型.

执行和异常总是要同时。如果您正在打开的文件不存在，那么，如果你没有正确处理这种情况，那么你的程序被认为是质量很差.

如果发生异常，程序停止。所以异常被用来处理各种不同类型的程序的执行过程中可能出现的错误，并采取适当的行动，而不是完全停止程序的.

Ruby提供了一个很好的机制来处理异常。我们附上的代码可以在开始/结束块引发的异常，并使用救援条款告诉Ruby我们要处理的异常类型.

语法 : 

begin  
# -  
rescue OneTypeOfException  
# -  
rescue AnotherTypeOfException  
# -  
else  
# Other exceptions
ensure
# Always will be executed
end

一切从开始拯救保护。如果在执行这个代码块中发生异常，控制传递给救援和结束之间的块.

对于每一个BEGIN块在拯救子句, Ruby比较依次对每一个参数引发的异常。匹配将成功拯救子句中命名的例外是如果目前抛出的异常的类型相同，或者是该异常的父类.

在异常不匹配任何指定的错误类型的事件后，我们被允许使用所有拯救子句的else子句.

例子: 

#!/usr/bin/ruby

begin
   file = open("/unexistant_file")
   if file
      puts "File opened successfully"
   end
rescue
      file = STDIN
end
print file, "==", STDIN, "\n"

这将产生以下结果。你可以看到，stdin是取代提交becauseopen失败. 

#==#

使用 retry 语句:

您可以使用拯救块捕获一个异常，然后使用重试要执行的语句从开头开始块.

语法: 

begin
    # Exceptions raised by this code will 
	# be caught by the following rescue clause
rescue
    # This block will capture all types of exceptions
    retry  # This will move control to the beginning of begin end

例子: 

#!/usr/bin/ruby

begin
   file = open("/unexistant_file")
   if file
      puts "File opened successfully"
   end
rescue
   fname = "existant_file"
   retry
end

以下是流程的过程:

  • 发生异常时开放
  • 前去营救, fname参数重新分配
  • 通过试到开头begin
  • 这次文件成功打开
  • 持续的基本过程.

注: 如果文件名称重新取代不存在这个例子的代码重试次数无限。要小心，如果您使用异常处理重试.

使用 raise 语句:

你可以使用raise语句抛出一个异常。下面的方法将引发异常时，它被触发调用。它的第二条消息将被打印出来. 

语法: 

raise 

OR

raise "Error Message" 

OR

raise ExceptionType, "Error Message"

OR

raise ExceptionType, "Error Message" condition

第一种形式简单reraises当前异常（或一个RuntimeError，如果没有当前异常）。这是用在需要拦截异常的异常处理之前将它传递.

第二种形式创建一个新的RuntimeError异常，给定的字符串设置它的消息。此异常然后提出了调用堆栈.

第三种形式中使用的第一个参数创建一个异常，然后第二个参数设置相关的消息。
第四种形式第三种形式是类似的，但你可以添加任何条件语句likeunless引发异常.

例子: 

#!/usr/bin/ruby

begin  
    puts 'I am before the raise.'  
    raise 'An error has occurred.'  
    puts 'I am after the raise.'  
rescue  
    puts 'I am rescued.'  
end  
puts 'I am after the begin block.'

这将产生以下结果: 

I am before the raise.  
I am rescued.  
I am after the begin block.

一个例子显示提高使用率: 

#!/usr/bin/ruby

begin  
  raise 'A test exception.'  
rescue Exception => e  
  puts e.message  
  puts e.backtrace.inspect  
end

这将产生以下结果: 

A test exception.
["test.rb:4"]

使用 ensure 语句:

有时你需要做一些处理，保证在一个代码块结束时，无论是否引发异常。例如，你可能有一个文件打开进入块，你需要确保它被封闭块退出.

ensure子句不只是这一点。确保后，最后的救援子句和包含块终止，将永远被执行的代码块。不要紧，如果退出该块通常，如果它引发和救援异常，或如果它是由一个未捕获的异常终止。ensure块将得到运行.

语法: 

begin 
   #.. process 
   #..raise exception
rescue 
   #.. handle error 
ensure 
   #.. finally ensure execution
   #.. This will always execute.
end

例子: 

begin
  raise 'A test exception.'
rescue Exception => e
  puts e.message
  puts e.backtrace.inspect
ensure
  puts "Ensuring execution"
end

这将产生以下结果: 

A test exception.
["test.rb:4"]
Ensuring execution

使用 else 语句:

如果else子句存在，它到rescue子句之后和之前的任何保证.

一个else子句的身体也不例外，只有通过代码的主体提出执行.

语法: 

begin 
   #.. process 
   #..raise exception
rescue 
   # .. handle error
else
   #.. executes if there is no exception
ensure 
   #.. finally ensure execution
   #.. This will always execute.
end

例子: 

begin
 # raise 'A test exception.'
 puts "I'm not raising exception"
rescue Exception => e
  puts e.message
  puts e.backtrace.inspect
else
   puts "Congratulations-- no errors!"
ensure
  puts "Ensuring execution"
end

这将产生以下结果: 

I'm not raising exception
Congratulations-- no errors!
Ensuring execution

可以使用$!变量捕获引发的错误消息.

Catch 和 Throw:

虽然除了raise和rescue机制，是伟大的放弃执行出问题时，它有时很高兴能够跳出正常的处理过程中一些深嵌套结构。这是捕获并抛出派上用场.

在catch定义一个给定的名称（这可能是一个符号或一个字符串）标记的块。块被正常执行，直到遇到一个throw.

语法: 

throw :lablename
#.. this will not be executed
catch :lablename do
#.. matching catch will be executed after a throw is encountered.
end

OR

throw :lablename condition
#.. this will not be executed
catch :lablename do
#.. matching catch will be executed after a throw is encountered.
end

例子:

下面的例子使用了抛出终止如果用户的互动“！”回应任何提示输入. 

def promptAndGet(prompt)
   print prompt
   res = readline.chomp
   throw :quitRequested if res == "!"
   return res
end

catch :quitRequested do
   name = promptAndGet("Name: ")
   age = promptAndGet("Age: ")
   sex = promptAndGet("Sex: ")
   # ..
   # process information
end
promptAndGet("Name:")

这将产生以下结果: 

Name: Ruby on Rails
Age: 3
Sex: !
Name:Just Ruby

Exception类:

Ruby的标准类和模块引发异常。所有的异常类，形成一个层次，在顶部的类异常。包含7种不同类型的一个新的水平:

  • Interrupt（打断）
  • NoMemoryError
  • SignalException
  • ScriptError
  • StandardError
  • SystemExit

在这个级别的，致命的是一个例外，但Ruby解释器只使用内部。
既ScriptError和StandardError有一个子类的数量，但我们并不需要去到的细节在这里。重要的是，如果我们创建自己的异常类，他们需要的任一类异常或其后裔之一的子类.


让我们看一个例子: 

class FileSaveError < StandardError
   attr_reader :reason
   def initialize(reason)
      @reason = reason
   end
end

现在看看下面的例子将使用此异常: 

File.open(path, "w") do |file|
begin
    # Write out the data ...
rescue
    # Something went wrong!
    raise FileSaveError.new($!)
end
end

这里最重要的行是：引发FileSaveError.new（$！）。我们呼吁提高信号发生异常，传递给它的存在的理由FileSaveError的一个新实例，具体的异常造成的数据写入失败.
